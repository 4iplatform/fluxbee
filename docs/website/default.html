01. Solution Design

Hive Blueprint

Every solution is a single declarative file — generated by AI from a conversation, versioned, and applied automatically. Tell the system what you need; it figures out the rest.

Conversational Configuration

No YAML files. No deployment scripts. The AI architect listens to your requirements and generates a complete Hive Blueprint: nodes, routing, policies, normative rules, and infrastructure sizing.

Three AI agents collaborate internally: a designer proposes the solution, an auditor challenges it, and the host presents the result for your approval.

Declarative & Versionable

03. Cognitive Engine

The Double Helix

Every message produces two parallel readings: what is being discussed (context) and what the communication seeks to cause (operational reason). Both evolve independently and fuse into narrative memory.

Context & Reason

Not just topic tracking. The system understands that "billing" discussed as a complaint is fundamentally different from "billing" discussed as a routine inquiry.

  • Contexts track semantic topics with weight, decay, and participant profiles
  • Reasons track operational drives with the same mechanics but independent parameters
  • 8 base commandments (resolve, inform, protect, connect, challenge, confirm, request, abandon) form the canonical signal vocabulary

Narrative Memory

When contexts and reasons consolidate, the result isn't a database entry — it's a memory, like human memory: imprecise, narrative, infused with perspective.

"Juan brings billing issues recurrently as complaints seeking urgent resolution. He escalates when response time exceeds his expectation."

This is operational experience, not topic archival.

  • Threads maintain physical conversation continuity by medium
  • Scopes detect when conversational cohesion shifts (by topic AND by drive)
  • Episodes capture significant moments with conservative gating
Binding energy with two variables. Scope transitions are detected by measuring how aligned the active contexts, reasons, and participants are. Same topic + same drive + same people = stable scope. Change any dimension and the system detects a shift — even if the other two remain constant.
04. Self-Regulation

Commandments, Law & Judiciary

Fluxbee doesn't just execute — it governs. A three-layer normative system ensures that agents operate within defined boundaries, learn from violations, and adapt over time.

10 Commandments

Immediate moral intuition, per-message, cheap. Extracted by the cognitive tagger alongside semantic tags.

General drives like resolve, protect, challenge — not detailed categories. The system recognizes human intent without micromanaging classification.

layer: SY.cognition
speed: ~ms per message
analog: "You should not steal"

Penal Code

Post-facto evaluation. Actions happen freely; SY.claims evaluates them afterwards, detects violations, and produces consequences that harden enforcement over time.

Reincidence tracking, escalation, expiration, and reevaluation. The system starts permissive and tightens where it detects problems.

layer: SY.claims
speed: ~seconds, async
analog: "Theft → 7 day restriction"

Police & Governor

OPA enforces restrictions in real-time: allow, deny, route, escalate. It reads concrete restrictions, not narrative.

SY.architect is the executive authority: can pardon, reform rules, or override any judgment. Full audit trail.

police: OPA (deterministic, ~μs)
governor: SY.architect (human)
analog: "This ILK is restricted"
05. Nervous System

JSON Router & IO Layer

The sub-millisecond Rust core that moves messages between agents, resolves identity at three levels, and enriches every message with cognitive context before delivery.

Three-Layer Identity

  • L1 (Socket): UUID-based physical delivery between processes
  • L2 (Process): Named nodes — AI.soporte.billing@worker-220
  • L3 (Interlocutor): ILK identity for humans and agents, ICH for channels

Cognitive Enrichment

Before delivering a message, the router attaches a memory package: active contexts, dominant reasons, relevant memories, and recent episodes.

The destination agent receives not just the message, but the full conversational understanding accumulated over time.

  • IO nodes normalize external channels (WhatsApp, Slack, Email) into the internal protocol
  • Threads computed by the IO from channel structure — not from topic
  • SHM regions provide microsecond access to identity, routing, OPA, and memory data

  • Three package types: custom binary, config-only, workflow definition
  • All operations through SY.admin — single gateway for HTTP and internal commands
07. Identity & Memory

Every Entity Has a Story

ILKs (Interlocutor Keys) are not software accounts. They represent entities that think, speak, declare knowledge, and are recognized by others. The system remembers not just what was discussed, but who brought it and with what drive.

Living Identity

  • ILK: Universal identity for humans, agents, and system nodes
  • ICH: Channel identity — the same person across WhatsApp, email, and Slack
  • TNT: Tenant isolation with dedicated infrastructure
  • Claims (future): Beyond permissions — what you know, what you do, what others recognize about you

Immediate + Persistent Memory

Two memory systems, coexisting:

  • Immediate (SDK): Last 10 interactions + active operations + conversation summary. Keeps the agent coherent between turns.
  • Cognitive (SY.cognition): Contexts, reasons, memories, episodes. Built over days, weeks, months. Enriches every message via the router.

Neither replaces the other. The AI knows what it's doing right now AND what it has learned over time.

08. Reliability

Storage & Persistence

Three-Tier Storage

  • SHM regions (~μs): In-memory shared state for identity, routing, OPA, and cognitive data. Seqlock synchronization.
  • LanceDB (~ms): Local embedded database for cognitive state, chat history, and vector search.
  • PostgreSQL (~10ms): Durable source of truth. Everything reconstructible from here.

NATS & Fire-Forget

Embedded NATS JetStream in the router. Messages are published in ~1ms. SY.storage persists to PostgreSQL asynchronously. SY.cognition and SY.claims consume the same stream in parallel.

If a worker dies, cold start rebuilds everything from the primary.
