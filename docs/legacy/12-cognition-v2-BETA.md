## 1. Purpose

SY.cognition is the cognitive engine of Fluxbee. It processes every message that flows through the system and builds structured understanding over time: what topics are being discussed, what has been learned, and what significant events have occurred.

Its responsibilities are:

1. **Tag messages** — extract semantic concepts from each message.
2. **Maintain contexts** — track active topics with weight, decay, and ILK participation.
3. **Detect scope transitions** — identify when the conversational cohesion shifts.
4. **Generate memories** — produce durable summaries of what the system has learned.
5. **Detect episodes** — identify significant socio-emotional events with a conservative gate.
6. **Serve cognitive data** — make contexts, memories, and episodes available for message enrichment.

SY.cognition does NOT handle:

- Immediate conversation memory (that's the AI SDK's responsibility — see `immediate-conversation-memory-spec.md`).
- Identity management (SY.identity).
- Message persistence to PostgreSQL (SY.storage).
- Routing decisions (router + OPA).

## 2. The Five Core Entities

### 2.1 Thread

The physical unit of conversation. Created by the IO node.

**Definition:** A thread is a group of ILKs communicating through a group of ICHs. It represents a concrete conversation channel: a WhatsApp group, a Slack channel, an email thread, a support ticket.

**Identity:** `thread:<hash>` where hash is deterministic based on the participants and channels. The IO node computes this because it knows the external channel structure.

**Rules:**

- The IO node creates the thread_id on first message if it doesn't exist.
- Same participants + same channel = same thread.
- Thread is stable: it doesn't change unless participants or channels change.
- All messages, contexts, memories, and episodes are associated with a thread.

**Who creates it:** The IO node. It knows the channel (ICH) and the participants (ILKs). It computes `thread_id = hash(sorted(ilk_ids) + sorted(ich_ids))` and includes it in the message metadata.

### 2.2 Scope

The abstract unit of conversational continuity. Emerges from accumulated context.

**Definition:** A scope groups threads that share history. When a new thread starts, the system checks if an existing scope matches (by participant overlap or ICH overlap) and associates it. This allows transferring contexts, memories, and episodes to a new thread.

**Identity:** `scope:<uuid-v4>`. Generated by SY.cognition when enough context accumulates to identify a coherent conversational space.

**Rules:**

- A scope does not appear immediately. With a single message there is no scope.
- A scope emerges when the binding energy model detects stable cohesion between contexts.
- A thread can be associated with at most one scope at a time.
- Contexts, memories, and episodes carry both `thread_id` and `scope_id` (if exists).
- When a new thread starts, SY.cognition checks for matching scopes by ILK overlap and associates if found.

**Scope Instance:** A temporal window within a scope. When binding energy indicates the conversational cohesion has shifted and this shift is sustained, the current scope instance closes and a new one begins. A scope can have many instances over time.

### 2.3 Context

A semantic topic tracked over time within a thread.

**Definition:** A context represents an active topic of conversation: "billing dispute", "technical support", "onboarding process". It accumulates weight as messages reinforce it and decays when not mentioned.

**Identity:** `context:<uuid-v4>`. Generated by SY.cognition when the context evaluator identifies a new topic.

**Mode:** Additive only. Existing contexts do not refine their label automatically. If a subtopic emerges, a new context is created. Contexts are never merged.

**Fields:**

```json
{
  "context_id": "context:uuid-v4",
  "thread_id": "thread:hash",
  "scope_id": "scope:uuid-v4",
  "scope_instance_id": "uuid-v4",
  "label": "billing dispute",
  "status": "open",
  "score": 7.8,
  "weight": 4.2,
  "weight_avg_cumulative": 2.85,
  "weight_avg_ema": 2.85,
  "weight_samples": 23,
  "tags": ["billing", "refund", "invoice", "payment"],
  "ilk_weights": {
    "ilk:uuid-juan": 6.0,
    "ilk:uuid-agent": 4.0
  },
  "ilk_profile": {
    "ilk:uuid-juan": { "as_sender": 12, "as_receiver": 3 },
    "ilk:uuid-agent": { "as_sender": 3, "as_receiver": 12 }
  },
  "opened_at": "2026-03-14T10:00:00Z",
  "last_seen_at": "2026-03-14T10:45:00Z",
  "closed_at": null
}
```

**Lifecycle:**

- Created when context evaluator identifies a new topic (score > CONTEXT_SCORE_THRESHOLD).
- Updated on each message that touches it (weight increases or decreases based on score).
- Closed when weight drops to 0 or below, or when MAX_OPEN_CONTEXTS guardrail triggers.

### 2.4 Memory

A durable summary of what the system has learned.

**Definition:** A memory is an extracted summary from one or more dominant contexts over a period. It represents condensed knowledge: "Juan has a recurring billing dispute with invoice FAC-2026-001" or "The onboarding flow for Acme Corp requires document verification".

**Identity:** `memory:<uuid-v4>`.

**Fields:**

```json
{
  "memory_id": "memory:uuid-v4",
  "thread_id": "thread:hash",
  "scope_id": "scope:uuid-v4",
  "scope_instance_id": "uuid-v4",
  "summary": "Juan has a recurring billing dispute regarding double charges on invoices FAC-2026-001 and FAC-2026-003.",
  "weight": 2.5,
  "occurrences": 3,
  "ilk_weights": {
    "ilk:uuid-juan": 5.0,
    "ilk:uuid-agent": 3.0
  },
  "created_at": "2026-03-14T10:00:00Z",
  "last_seen_at": "2026-03-14T14:00:00Z"
}
```

**Rules:**

- Created by the scope summarizer AI when analyzing a period.
- Reuses existing memories when AI identifies a match (reinforcement, not duplication).
- Weight increases on reinforcement, decays when not reinforced (`MEMORY_DECAY_STEP = 0.25`).
- Summary is a single descriptive paragraph, not a keyword or tag.

### 2.5 Episode

A significant socio-emotional event.

**Definition:** An episode is a noteworthy event detected during a conversation: friction, frustration, delight, escalation, complaint. It differs from a context (which is a topic) and from a memory (which is knowledge). An episode captures a moment of emotional or relational significance.

**Identity:** `episode:<uuid-v4>`.

**Fields:**

```json
{
  "episode_id": "episode:uuid-v4",
  "thread_id": "thread:hash",
  "scope_id": "scope:uuid-v4",
  "scope_instance_id": "uuid-v4",
  "period_index": 2,
  "period": { "start_ts": "...", "end_ts": "..." },
  "main_context": { "context_id": "context:uuid", "label": "billing dispute" },
  "memory_id": "memory:uuid-v4",
  "fear_id": "anger",
  "title": "Friction over delayed refund",
  "summary": "Customer expressed strong frustration about refund processing time.",
  "subthemes": ["complaint_about_time", "hostile_tone"],
  "status": "pending",
  "base_intensity": 8,
  "novelty_bonus": 0,
  "habit_count": 2,
  "habit_penalty": 0,
  "intensity": 8,
  "reason": "Explicit friction evidence in dominant billing context",
  "evidence_context_ids": ["context:uuid"],
  "evidence_signals": ["anger", "complaint"],
  "evidence_strength": 9
}
```

**Gate (conservative):** Episodes are only persisted when they pass a multi-condition gate:

- `base_intensity >= EPISODE_MIN_FINAL_INTENSITY (7)`
- `evidence_strength >= EPISODE_MIN_EVIDENCE_STRENGTH (8)`
- Evidence must be anchored to a dominant context.
- Ambiguous episodes are NOT persisted (logged for audit but not stored).

## 3. Pipeline

### 3.1 Overview

Every message that flows through the system is processed by SY.cognition:

```
Message arrives (via NATS from router)
  │
  ▼
1. TAGGER
   Extract semantic tags from the message.
   Input: message text + word_count
   Output: tags[]
  │
  ▼
2. CONTEXT EVALUATOR
   Assign message to existing contexts or create new ones.
   Input: tags + open contexts + identity metadata
   Output: assignments, new_contexts, closures
   Mode: tags-only (no raw text), additive only
  │
  ▼
3. DETERMINISTIC CONTEXT UPDATE
   Update weights, ILK profiles, decay untouched contexts.
   Apply MAX_OPEN_CONTEXTS guardrail.
   Emit context events.
  │
  ▼
4. SCOPE BINDING EVALUATION
   Calculate binding energy between active contexts.
   Update EMA, evaluate threshold, detect scope change.
   Emit scope events if instance changes.
  │
  ▼
5. PERIOD DETECTION (when scope instance changes or periodically)
   Build snapshots by timestamp with main context.
   Cut periods by switch, decay, or time threshold.
  │
  ▼
6. SCOPE SUMMARIZER (per period)
   Generate memories and evaluate episodes.
   Input: period data + existing memories + contexts + fears
   Output: period_memories + global_adjustments + episode_decision
   Single AI call for both memories and episodes.
  │
  ▼
7. APPLY RESULTS
   Create/update memories, apply decay to unreinforced ones.
   Persist episodes that pass the gate.
   Update LanceDB + jsr-memory SHM.
  │
  ▼
8. PUBLISH TO NATS (fire & forget)
   Publish events, items, reactivations for SY.storage to persist in PostgreSQL.
```

### 3.2 Where Cognitive Gets Messages

SY.cognition consumes messages from NATS (`storage.turns` stream). It does NOT receive messages directly from the router. This is the same stream that SY.storage consumes.

The router publishes every non-system message to NATS as fire & forget (~1ms). SY.cognition subscribes to the same stream and processes each message through the pipeline.

### 3.3 Message Enrichment

When a message is being routed, the router reads `jsr-memory-<hive>` SHM to attach relevant cognitive data to the message before forwarding it to the destination node. This is the `memory_package` in the message meta.

The router enriches, not SY.cognition. SY.cognition only writes the data; the router reads and attaches it.

## 4. AI Contracts

### 4.1 Tagger

**Input:**

```json
{
  "word_count": 18,
  "max_tags": 12,
  "text": "Message text"
}
```

**Output:**

```json
{
  "tags": ["billing", "refund", "complaint"]
}
```

**Post-processing rules:**

- Lowercase, no punctuation/symbols.
- Max 3 words per tag.
- Deduplicate preserving order.
- Final limit: `tags.length <= min(word_count, MAX_TAGS)`.

Each tag carries metadata from the message:

```json
{
  "tag": "billing",
  "ilk_sender": "ilk:uuid-juan",
  "ilk_receiver": "ilk:uuid-agent"
}
```

### 4.2 Context Evaluator

**Input:**

```json
{
  "CONTEXT_SCORE_THRESHOLD": 4,
  "tags": [{"value": "billing"}, {"value": "refund"}],
  "open_contexts": [
    {
      "context_id": "context:uuid",
      "label": "billing dispute",
      "score": 7.2,
      "weight": 4.1,
      "ilk_weights": {"ilk:uuid-juan": 6.0},
      "ilk_profile": {"ilk:uuid-juan": {"as_sender": 4, "as_receiver": 1}},
      "weight_avg_cumulative": 2.85,
      "weight_avg_ema": 2.85,
      "weight_samples": 23,
      "status": "open",
      "opened_at": "...",
      "last_seen_at": "..."
    }
  ],
  "identity": {
    "ich_id": "ich:uuid",
    "sender_ilk": "ilk:uuid",
    "recipient_ilks": ["ilk:uuid"]
  }
}
```

**Output:**

```json
{
  "assignments": [
    {"context_id": "context:uuid", "label": "billing dispute", "score": 7.8}
  ],
  "new_contexts": [
    {"label": "refund process", "score": 8.3}
  ],
  "to_close": [
    {"context_id": "context:uuid-old", "reason": "topic finished"}
  ]
}
```

**Rules:**

- Only items with `score > CONTEXT_SCORE_THRESHOLD` are accepted.
- `new_contexts` do not carry `context_id` (generated by backend).
- `assignments` must reference existing contexts by `context_id`.
- Mode is additive: do not refine existing labels, create new context for subtopics.

### 4.3 Scope Summarizer

Single AI call that produces both memories and episode decisions per period.

**Input:**

```json
{
  "period": {"start_ts": "...", "end_ts": "..."},
  "main_context": {"context_id": "context:uuid", "label": "billing dispute"},
  "memories": [
    {"memory_id": "memory:uuid", "summary": "...", "weight": 2.5, "occurrences": 3}
  ],
  "fears": [
    {"fear_id": "anger", "label": "Anger / Friction", "description": "..."}
  ],
  "prior_stats": [
    {"fear_id": "anger", "main_context_id": "context:uuid", "count": 2}
  ],
  "contexts": [
    {
      "context_id": "context:uuid",
      "label": "billing dispute",
      "weight": 4.2,
      "weight_avg_cumulative": 2.9,
      "weight_avg_ema": 2.9
    }
  ]
}
```

**Output:**

```json
{
  "period_memories": [
    {
      "extracto": "Customer has ongoing billing dispute about double charges.",
      "memory_match_id": "memory:uuid",
      "weight_delta": 1.2,
      "enrich": true
    }
  ],
  "global_adjustments": [
    {"memory_id": "memory:uuid-old", "weight_delta": -0.5, "reason": "lost relevance"}
  ],
  "episode_decision": {
    "should_create_episode": true,
    "fear_id": "anger",
    "title": "Friction over delayed refund",
    "summary": "...",
    "subthemes": ["complaint_about_time", "hostile_tone"],
    "status": "pending",
    "base_intensity": 8,
    "novelty_bonus": 0,
    "evidence_context_ids": ["context:uuid"],
    "evidence_signals": ["anger", "complaint"],
    "evidence_strength": 9,
    "reason": "Explicit friction in dominant context"
  }
}
```

**Rules:**

- Prioritize reusing existing memories over creating new ones.
- Avoid duplicate memories for the same period.
- Summary is a single descriptive paragraph.
- If friction is explicit, do not negate it narratively.
- When ambiguous: `should_create_episode = false`.

## 5. Deterministic Algorithms

### 5.1 Context Update Per Message

1. Resolve assignments and new contexts from evaluator.
2. For each open context:
   - Calculate `touched` (was it assigned or mentioned).
   - Calculate weight delta based on score (additive mode).
   - Update `weight`.
   - Update `weight_avg_cumulative` and `weight_avg_ema`.
3. If `weight <= 0`, close context.
4. If touched and identity present:
   - Accumulate ILK weights (`ILK_CONTEXT_WEIGHT_SENDER = 1.0`, `ILK_CONTEXT_WEIGHT_RECIPIENT = 0.5`).
   - Update `ilk_profile` (as_sender / as_receiver counts).
5. Apply `MAX_OPEN_CONTEXTS` guardrail (20):
   - If exceeded, sort by score desc, keep top 20.
   - Close excess with event `CONTEXT_ENDED` reason `max_open_contexts`.

### 5.2 Scope Binding Energy

Executed after context update on each message. Only when `active contexts >= SCOPE_ENERGY_MIN_CTX_FOR_EVAL (2)`.

**Step 1 — Tag similarity (Jaccard):**

```
S_tema(i,j) = |tags_i ∩ tags_j| / |tags_i ∪ tags_j|
```

**Step 2 — ILK similarity (Cosine):**

```
vec_i[ilk] = ilk_profile[ilk].as_sender + ilk_profile[ilk].as_receiver
S_ilk(i,j) = dot(vec_i, vec_j) / (|vec_i| × |vec_j|)
```

**Step 3 — Coupling:**

```
J(i,j) = S_tema(i,j) × S_ilk(i,j)
```

**Step 4 — Energy:**

```
E = Σ -J(i,j) × ema_i × ema_j    (for all pairs i < j)
```

**Step 5 — Normalize:**

```
M = Σ ema_i    (sum of all active context EMAs)
Ê = E / (M × M)
```

Range: approximately -1 (strongly coupled) to 0 (decoupled).

**Step 6 — Smooth with EMA:**

```
energy_ema = energy_ema + ALPHA × (Ê - energy_ema)
SCOPE_ENERGY_ALPHA = 0.25
```

**Step 7 — Evaluate threshold:**

```
if energy_ema > SCOPE_ENERGY_UNBIND_THRESHOLD (-0.05):
    unbind_streak += 1
else:
    unbind_streak = 0

if unbind_streak >= SCOPE_ENERGY_SUSTAIN_COUNT (2):
    → scope instance changed
```

### 5.3 Memory Period Detection

1. Build snapshots by timestamp with main context per snapshot.
2. Cut periods by:
   - Sustained context switch.
   - Strong single context switch.
   - Strong decay of main context.
3. Apply anti-noise filter (ignore very short periods).

### 5.4 Memory Application Per Period

1. AI returns `period_memories` + `global_adjustments`.
2. Apply global adjustments (weight changes to existing memories).
3. Apply period actions:
   - Match found → update weight + enrich summary if flagged.
   - No match → create new memory.
4. Apply decay to unreinforced memories (`MEMORY_DECAY_STEP = 0.25`).

### 5.5 Episode Gate

Per period:

1. Read `episode_decision` from AI.
2. Calculate final intensity with habit penalty.
3. Validate gate (all conditions must pass):
   - `should_create_episode == true`
   - `final_intensity >= EPISODE_MIN_FINAL_INTENSITY (7)`
   - `evidence_strength >= EPISODE_MIN_EVIDENCE_STRENGTH (8)`
   - Evidence anchored to dominant context (`has_dominant_context_evidence`).
   - Fear_id present and valid.
   - Summary is not empty.
4. Only persist if all conditions pass.
5. Update habit counter for `fear_id::main_context_id` if persisted.

## 7. Architecture

### 7.1 One Per Hive

SY.cognition runs on every hive (motherbee and workers). Each instance processes messages from its local NATS stream.

### 7.2 Data Flow

```
Router
  │ publish to NATS (storage.turns)
  ▼
SY.cognition (local)
  │ reads from NATS
  │ processes pipeline
  │ writes LanceDB (local)
  │ writes jsr-memory SHM (local)
  │ publishes to NATS (storage.events, storage.items, storage.reactivation)
  ▼
SY.storage (motherbee only)
  │ reads from NATS
  │ writes PostgreSQL
  ▼
PostgreSQL (source of truth)
```

### 7.3 Storage Layers

| Layer | Purpose | Location | Latency |
|-------|---------|----------|---------|
| jsr-memory SHM | Index for router enrichment (tags → events/items) | `/dev/shm/jsr-memory-<hive>` | ~μs |
| LanceDB | Local cognitive database (contexts, memories, episodes, vectors) | `/var/lib/fluxbee/nodes/SY/SY.cognition@<hive>/memory.lance` | ~ms |
| PostgreSQL | Durable source of truth (via SY.storage) | Motherbee only | ~10ms |

Active threads: in LanceDB + SHM.
Inactive threads: in PostgreSQL (reconstructible via cold start).

### 7.4 Cold Start

When SY.cognition starts with empty LanceDB:

1. Connect to SY.storage via socket.
2. Request historical turns for the hive.
3. Replay through pipeline to rebuild contexts, memories, episodes.
4. Write to LanceDB + jsr-memory SHM.
5. Begin processing live NATS stream.

### 7.5 SHM Region: jsr-memory-\<hive\>

Single writer: SY.cognition. Uses seqlock (same as all other regions).

Contains inverted index of tags → event/memory references for fast lookup by router during message enrichment.

The router reads this region to build the `memory_package` that is attached to messages before delivery.

## 8. Thread Lifecycle

### 8.1 Creation

```
1. IO receives message from external channel (e.g., WhatsApp).
2. IO computes thread_id = hash(sorted(ilk_ids) + sorted(ich_ids)).
3. IO includes thread_id in message metadata.
4. Message goes through router to destination + NATS.
5. SY.cognition receives message from NATS.
6. If thread_id is new: create thread record in LanceDB.
7. Process message through cognitive pipeline.
```

### 8.2 Thread in Message Meta

```json
{
  "meta": {
    "thread_id": "thread:a1b2c3d4...",
    "src_ilk": "ilk:uuid-juan",
    "dst_ilk": "ilk:uuid-agent",
    "ich": "ich:uuid-whatsapp"
  }
}
```

### 8.3 Scope Association

When a new thread appears, SY.cognition checks for matching scopes:

1. Extract ILKs from the thread.
2. Search existing scopes for ILK overlap (same participants in different threads).
3. If match found: associate thread with existing scope → load scope's contexts, memories, episodes.
4. If no match: thread starts without scope. Scope emerges later from accumulated context.

## 10. Events Published to NATS

SY.cognition publishes to NATS for SY.storage to persist in PostgreSQL:

### 10.1 Context Events

Subject: implicit in the context data published.

Types: `CONTEXT_STARTED`, `CONTEXT_UPDATED`, `CONTEXT_ENDED`, `CONTEXT_ILK_WEIGHTS_UPDATED`, `SCOPE_ENERGY_UPDATED`, `SCOPE_CHANGED`.

### 10.2 Memory Events

Types: `MEMORY_CREATED`, `MEMORY_WEIGHT_UPDATED`.

Reasons: `ai_new_memory`, `ai_period_memory`, `ai_global_adjustment`, `memory_decay_not_reinforced`.

### 10.3 Episode Events

Published when an episode passes the gate and is persisted.

### 10.4 Reactivation Events

Published when a node AI uses a memory or episode (feedback loop for activation_strength adjustment).

All publishes are fire & forget to NATS. SY.storage consumes and writes to PostgreSQL with idempotent writes (`ON CONFLICT DO NOTHING`).

## 11. Naming: v1.16 to v2 Migration

| v1.16 Term | v2 Term | What Changed |
|------------|---------|--------------|
| `ctx:<hash>` (conversation ID) | `thread:<hash>` | Thread is the physical conversation unit, created by IO |
| Episode (block of turns between boundaries) | Scope Instance | Temporal window within a scope, cut by binding energy |
| Episode (as event) | Episode | Now specifically a socio-emotional event with intensity and gate |
| Event (episodic unit) | Context + Scope | Contexts track topics, scopes track cohesion |
| Memory Item (fact/procedure/constraint) | Memory | Narrative summary with weight and decay, not atomic facts |
| Highlight (verbatim turns) | (preserved in episode evidence) | Still exists as evidence within episodes |
| Cue/Tag | Tag | Simplified naming, same function |
| AI.tagger (external node) | Tagger (internal to SY.cognition pipeline) | Cognitive owns its own AI calls |
| AI.consolidator (external node) | Scope Summarizer (internal to SY.cognition pipeline) | Single AI call for memories + episodes |
| activation_strength | weight | Simplified naming, same decay/reinforcement concept |
| context_inhibition | (deferred) | Not needed in v2, may return later |
| jsr-memory (tag index) | jsr-memory (same) | Same region, updated schema for new entities |

## 12. Open Design Questions

These are explicitly deferred. They should be resolved during implementation, not in this spec:

### 12.1 Scope Association Algorithm

How exactly does SY.cognition match a new thread to an existing scope? By ILK overlap threshold? By ICH overlap? By both? What is the minimum overlap to associate?

### 12.2 Fears Catalog

The episode model references `fear_id` (anger, frustration, etc.). Where does the catalog of fears live? Options: hardcoded in SY.cognition, configurable in hive.yaml, administered by SY.admin. For v2 pilot, hardcoded is sufficient.

### 12.3 AI Call Strategy

Does SY.cognition make direct AI calls (like SY.architect), or does it send messages to AI nodes? For v2: direct calls using the AI SDK, same as architect. SY.cognition carries its own AI provider key in its config.

### 12.4 Feed/Stream Processing

Continuous feeds (video, audio, sensor data) are explicitly deferred. The current model assumes discrete messages. Continuous input requires a preprocessing layer (IO segmentation) before cognitive can process it. This is documented as a known gap.

### 12.5 Thread ID Computation Details

The exact hash function and input format for `thread_id = hash(sorted(ilk_ids) + sorted(ich_ids))` needs to be specified. Options: SHA-256 truncated, blake3, etc. The IO node SDK should provide a canonical `compute_thread_id()` function.

## 13. Relationship to Immediate Memory (AI SDK)

SY.cognition and the AI SDK's immediate memory serve different purposes:

| Aspect | Immediate Memory (AI SDK) | Cognitive Memory (SY.cognition) |
|--------|--------------------------|--------------------------------|
| Scope | Single chat session, single turn | Cross-session, cross-thread |
| Horizon | Last 10-12 interactions | Days, weeks, months |
| Owner | Each AI node | SY.cognition (system-wide) |
| Storage | In-memory per session | LanceDB + PostgreSQL |
| Purpose | Operational continuity (what am I doing right now) | Learned knowledge (what do I know about this entity/topic) |
| Consumed by | The AI node itself | Router (enrichment) + AI nodes (via memory_package) |

Both coexist. Immediate memory is necessary even after cognitive memory exists. Cognitive memory is additive — it enriches the AI's context with longer-range knowledge that immediate memory cannot retain.
