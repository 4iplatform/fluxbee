# Solution Manifest — Declarative Configuration per Tenant

**Status:** v1.0 (beta)
**Date:** 2026-03-31
**Audience:** SY.architect developers, orchestrator, admin API
**Parent specs:** `node-spawn-config-spec.md`, `runtime-packaging-cli-spec.md`, `sy-architect-spec.md`, `sy-policy-beta.md`

---

## 1. Purpose

A solution manifest is a single JSON file that describes the complete desired state of a Fluxbee solution for one tenant. It covers topology, runtimes, nodes, routing, OPA, normative rules, and identity — everything needed to go from an empty system to a running solution.

SY.architect generates this file conversationally, versions it, and applies it by reconciling the desired state against the actual system state.

---

## 2. Problem

Today, deploying a complete solution requires dozens of manual API calls: create tenant, add hives, publish runtimes, spawn nodes with individual configs, add routes, configure VPN, apply OPA policies, set up claims rules. There is no way to:

- Describe a solution declaratively.
- Reproduce it in another environment.
- Roll back to a known good state.
- Diff two versions and see what changed.
- Let the architect generate it from a conversation without the user knowing the internals.

---

## 3. Design Decisions

1. **One manifest per tenant-solution.** A tenant can have one solution manifest that describes everything running under it. If the tenant needs more capabilities (e.g., add marketing to an existing support system), the manifest is extended and re-versioned, not replaced by a second file.

2. **Single JSON format.** No dual format. JSON with clear naming and `description` fields on every block is readable by both LLMs and humans. The LLM generates and modifies the JSON directly.

3. **Declarative, not imperative.** The manifest says "this is the desired state." It does NOT say "do these steps." SY.architect reads the manifest, compares with actual state, and executes the minimum ADMIN_COMMANDs to converge.

4. **Tenant isolation at hive level.** Workers are dedicated per tenant. Mixing tenants on the same worker is explicitly avoided for simplicity and security. The manifest declares which hives belong to the tenant and what runs on each.

5. **Versioned.** Every change produces a new version. History is preserved. Rollback applies a previous version's manifest.

6. **Generated by architect's agent triunvirate.** The user describes what they want conversationally. The architect agent (host) coordinates with the programmer agent (designs the solution) and the auditor agent (validates it). The result is a solution manifest that the host applies.

## 5. Sections Explained

### 5.1 solution

Metadata: name, semver version, description, timestamps. Every modification bumps the version.

### 5.2 tenant

The tenant this solution belongs to. One manifest per tenant. If the tenant doesn't exist yet, architect creates it before applying the rest.

### 5.3 topology

Hives and network. `tenant_exclusive: true` means this worker serves only this tenant. Includes VPN definitions. Capacity hints help architect make sizing decisions.

### 5.4 runtimes

Packages that need to be published for this solution. Architect checks if they're already published (via `get_versions`) and publishes missing ones. References the three package types from `runtime-packaging-cli-spec.md`.

### 5.5 nodes

Concrete node instances to deploy. Each has a hive assignment, runtime reference, and config. The config can reference external assets via `system_prompt_ref` (resolved by orchestrator from the package's assets directory).

### 5.6 routing

Message routing rules. Describes how messages flow between nodes based on identity, context, and other criteria. Includes fixed routes for inter-node communication.

### 5.7 policy

The unified claims × actions → effects matrix. This is the law of the solution. Each rule says: "if an ILK with these claims performs this action, the effect is X." The matrix serves two purposes simultaneously: it compiles to OPA Rego for real-time enforcement, and SY.policy uses it as reference for post-facto evaluation. One source of truth, two consumers. See `sy-policy-beta.md` for the full model.

### 5.9 identity

Controlled vocabulary for this solution's domain. Initial ILKs for testing or pre-registration.

### 5.10 infrastructure_guidelines

Sizing recommendations and best practices. Not enforced — informational for the architect and the operator.

## 6. Architect's Agent Triunvirate

### 6.1 Roles

| Agent | Prompt file | Role in manifest generation |
|-------|------------|---------------------------|
| Host (architect) | `architect.md` | Talks to user, coordinates, applies manifest |
| Programmer | `programmer.md` | Designs the solution: picks runtimes, sizes topology, writes configs |
| Auditor | `auditor.md` | Validates: checks consistency, identifies gaps, challenges assumptions |

### 6.2 Generation Flow

```
User: "I need a customer support system for WhatsApp with billing and tech support"
  │
  ▼
Host receives request, forwards to Programmer
  │
  ▼
Programmer generates initial manifest:
  - Picks runtimes (ai.generic + config_only for agents)
  - Sizes topology (1 worker for this load)
  - Defines nodes, routing, OPA
  - Estimates load and sizing
  │
  ▼
Host forwards manifest to Auditor
  │
  ▼
Auditor reviews:
  - "Missing escalation path for failed billing resolution"
  - "No VPN configured between hives"
  - "OPA missing tenant isolation rule"
  - Score: 7/10
  │
  ▼
Host sends feedback to Programmer
  │
  ▼
Programmer revises manifest (adds escalation WF, VPN, OPA rule)
  │
  ▼
Auditor re-reviews:
  - "Looks complete. Minor: no rate limit on WhatsApp webhook"
  - Score: 9/10
  │
  ▼
Host presents to user:
  "Here's the proposed solution: 2 support agents (billing + tech),
   WhatsApp integration, escalation workflow. 1 dedicated worker.
   Shall I deploy it?"
  │
  ▼
User: "Yes"
  │
  ▼
Host applies manifest via reconciliation
```

### 6.3 Scoring

The auditor scores each manifest version (1-10). The threshold for auto-approval is configurable (default: 8). Below threshold, the programmer must revise. Above threshold, the host can present to the user.

## 7. Reconciliation (Apply)

### 7.1 Concept

When architect applies a manifest, it does NOT execute it as a script. It reconciles: compares desired state (manifest) with actual state (system) and applies only the differences.

### 7.2 Reconciliation Order

```
1. TENANT
   - Check if tenant exists → create if missing
   - Update tenant metadata if changed

2. TOPOLOGY
   - Check hives exist → add_hive for missing ones
   - Check VPNs → add_vpn for missing ones
   - Check network configuration

3. RUNTIMES
   - Check published runtimes → fluxbee-publish for missing ones
   - Check versions → update if newer version in manifest
   - Sync to workers → sync-hint + update

4. NODES
   - Compare desired nodes with actual nodes
   - Spawn missing nodes with config
   - Update config of existing nodes if changed
   - Kill nodes that are in actual but not in manifest (with confirmation)

5. ROUTING
   - Compare desired routes with actual routes
   - Add missing routes
   - Remove routes not in manifest (with confirmation)

6. POLICY
   - Compile policy_matrix to Rego
   - Apply via opa_apply → jsr-opa SHM
   - Load matrix into SY.policy as evaluation reference
   - Verify OPA rules are active

7. IDENTITY
   - Add vocabulary terms
   - Create initial ILKs if specified
```

### 7.3 Safety Rules

- Destructive actions (kill nodes, remove routes, remove hives) always require user confirmation.
- Reconciliation is idempotent: applying the same manifest twice produces no changes.
- If a step fails, reconciliation stops and reports the partial state to the user.

## 8. Versioning

### 8.1 Version Format

Semver: `MAJOR.MINOR.PATCH`

- MAJOR: breaking changes (new hives, removed nodes, topology change).
- MINOR: new capabilities (new node, new route, new rule).
- PATCH: config tweaks (prompt change, threshold adjustment).

### 8.2 History

Architect stores manifest history in LanceDB:

```json
{
  "version": "2.1.0",
  "manifest": { ... },
  "applied_at": "2026-03-14T14:00:00Z",
  "applied_by": "ilk:uuid-architect",
  "changes_summary": "Added escalation workflow, updated billing prompt",
  "reconciliation_log": [
    { "action": "run_node", "target": "WF.escalation.acme", "status": "ok" },
    { "action": "set_node_config", "target": "AI.soporte.billing.l1", "status": "ok" }
  ]
}
```

### 8.3 Rollback

To rollback: architect loads a previous version's manifest and applies it via reconciliation. The system converges to the previous state. New nodes in the current version are killed (with confirmation). Removed nodes in the previous version are respawned.

### 8.4 Diff

Architect can diff two versions and show the user what changed:

```
v2.0.0 → v2.1.0:
  + Added node WF.escalation.acme@worker-acme-01
  ~ Updated AI.soporte.billing.l1 config (prompt, temperature)
  + Added route escalation-path
  + Added normative rule norm:100 (billing refund limit)
```

## 9. Secrets Management

API keys and credentials appear in the manifest (node configs, IO webhook tokens). For v1:

- Secrets are in plaintext in the manifest (same as node config.json).
- Manifest files should have restricted permissions (0600).
- Manifest history should be stored in the architect's local LanceDB, not in shared storage.

For future: secrets replaced by references (`secret://openai-key`) resolved at apply time from a secure store.
